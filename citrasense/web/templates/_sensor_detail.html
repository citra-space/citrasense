{# Sensor detail page, served for client-side path /sensors/. The outer wrapper resolves ``$store.citrasense.currentSensor`` into a local ``sensor`` alias via ``x-data="{ get sensor() {...} }"`` so every ``sensor.*`` binding inside the per-modality partials sees the current sensor without each partial having to wire its own alias. If the URL points at a deleted sensor, ``currentSensor`` is null, ``x-if`` skips rendering, and a watcher in the store bounces back to ``/monitoring``. Dispatch: the actual page body is a per-modality partial (``_sensor_detail_telescope.html``, ``_sensor_detail_radar.html``, or the generic placeholder below for unknown types). Each per-modality partial owns its own page header (name / type pill / hardware pills / optional switches) so the dispatcher stays modality-agnostic. #}
{# Stale-link guard: if we land on /sensors/ for a sensor that doesn't exist (deep link to a deleted scope, or the operator just removed this sensor from Config), currentSensor resolves to null and we bounce back to /monitoring. StatusCollector prunes status.sensors on every tick, so this triggers automatically. Critical: the guard must only fire *after* the first status payload has arrived. Otherwise a deep link / cmd-click / hard-refresh on /sensors/ bounces to /monitoring before the WebSocket has even delivered ``sensors``. Gating on ``sensors.length > 0`` is the simplest correct check — an empty sensor list means "still loading" here. A site that legitimately has zero sensors would just render the detail shell with no body, which is fine. #}
